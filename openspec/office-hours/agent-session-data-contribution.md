## Problem Statement

开源模型生态最稀缺的不是模型权重，而是**真实世界的人机协作开发轨迹数据**（agentic coding trajectories）。闭源实验室通过自家产品持续获得海量真实用户与 AI 结对编程的过程数据，而开源社区几乎只能依赖合成数据自举。与此同时，每个使用 AI 编程工具的开发者本地都沉睡着大量这类数据（如 Claude Code 存于 `~/.claude/projects/` 的完整会话 JSONL），却因为**隐私风险（密钥、公司代码、个人信息）没有安全的贡献通道**而无法流入社区。

本项目受梁文锋"科技平权、反垄断"理念启发：构建一个**会话脱敏导出工具 + 社区数据集**，让任何开发者都能安全地把自己的开发过程数据贡献给开源社区。

## What Makes This Cool

1. **打在开源生态最疼的软肋上**：真实 agent 轨迹数据是当前开源模型与闭源模型差距的核心来源之一，个人开发者第一次有了直接参与弥合这个差距的通道。
2. **结构感知脱敏是空白地带**：现有工具（Gitleaks、TruffleHog、Presidio）都面向代码仓库或平面文本；没有工具理解 agent 会话的结构（敏感信息集中在工具调用结果、命令回显、系统提示的特定位置）。结构感知管道可以比平面扫描准得多。
3. **工具即飞轮**：只贡献自己的数据是一次性的；工具让每个开发者都变成贡献者。
4. **双通道满足不同隐私偏好**：既有完全公开的社区数据集通道，也有"只有我和目标厂商能看到"的私密直投通道。

## Constraints

- 个人项目 / 开源项目，无预算养服务器（v1 阶段零基建）。
- 贡献者人群不只是开发者：code CLI 已相当普及，大量非开发者用户既没有 HuggingFace 账号、也未必有 GitHub 账号。GitHub PR 通道服务有账号的开发者主力人群；无 GitHub 账号的用户可走出口②（API 直投）参与贡献，自建服务器收件（v2）再进一步降低公开通道的门槛。
- 会话数据包含 API 密钥、公司代码、个人路径等，泄漏一次即摧毁项目信誉——脱敏是命门。
- API 直投通道使用贡献者自己的 key，产生真实 token 费用。

## Premises（已与发起人确认）

1. 项目核心产出是「数据 + 让更多人能贡献数据的工具」，不是模型或代码。
2. 脱敏检测器不自研：复用 Gitleaks 规则集等成熟引擎做清洗层；独特价值在「面向 agent 会话结构的管道 + 人工审查体验 + 统一导出 schema」。
3. 纯自动清洗在理论上不充分（研究表明写作风格/话题可重识别用户），因此「用户主动选择可公开范围 + 导出前强制人工确认」是默认设计，不是可选项。人工确认的精确语义：Layer 1/2（密钥与自定义规则）命中项**全量逐项处置、未清零不解锁**；Layer 3（归一化）为全量统计报告 + 高危片段抽检——用户签署的是"命中项已全部处置 + 抽检通过"，不承诺通读全部数据。
4. 数据集 schema/发布方案由发起人已有腹稿主导，工具导出格式对齐该方案（详见 Open Questions）。

## Approaches Considered

### Approach A: 直通投喂（发起人原方案）
- **Summary**: 拷贝本地会话 → 规则脱敏 → 环境变量注入（base_url + key 指向目标厂商，如 DeepSeek）→ 重放会话 + meta 消息，数据以正常 API 流量形式进入厂商服务器。
- **Effort**: S-M（若目标站点无 Anthropic 兼容端点，需额外做 API 格式转换层）| **Risk**: Med
- **Pros**: 最快上线；零发布基建；隐私链路最短（仅用户与目标厂商可见）。
- **Cons**: 只惠及一家厂商，偏离平权初心；无法验证数据是否被用于训练（普通推理流量可能被厂商数据管线当噪声去重）；每次重放消耗真实 token 费用。

### Approach B: 开放数据集管道
- **Summary**: 同样的脱敏核心 + 统一 schema 导出 + 人工确认后发布到 HuggingFace 公开数据集。
- **Effort**: M | **Risk**: Low
- **Pros**: 真正的平权（所有开源模型受益，包括 DeepSeek）；可验证、可累积；能形成社区飞轮。
- **Cons**: 需设计 schema；发布流程多一步；放弃了私密直投机制。

### Approach C: 双通道架构（✅ 已选定）
- **Summary**: pipeline = 采集 → 脱敏 → 人工审查 → 可插拔出口（出口① HF 公开数据集；出口② API 重放直投），用户按隐私偏好自选。
- **Effort**: M | **Risk**: Low-Med
- **Pros**: 一套脱敏核心同时满足私密直投机制与平权初心；出口插件化与采集端 adapter 理念自洽；v0.1 可先实现单个出口。
- **Cons**: 比 B 多一个出口的维护量；出口②的 ToS 风险仍存在（见 Open Questions）。
- **Reuses**: Gitleaks 规则集（密钥检测）、可选 Presidio/本地 LLM 层（PII 检测）、GitHub PR + CI 作为社区收集与二次审查基建。

## Recommended Approach

**Approach C：双通道架构**。整体数据流：

```
[采集层 - adapter 可插拔]
  v0.1: Claude Code 本地会话 JSONL (~/.claude/projects/)
  未来: Codex / Cursor / Cline 等 adapter
        │  用户主动选择可公开的项目/会话范围（白名单式）
        │  ★ 白名单范围选择是「专有代码不泄漏」的第一道也是主要防线：
        │    脱敏层抓的是密钥/PII，不负责判断代码本身可否公开。
        │    工具默认只推荐 git remote 为公开仓库的项目进入候选。
        ▼
[工作区拷贝]  原始数据只读，处理在隔离文件夹进行
        │  非文本内容（base64 图片、二进制附件）**标记而非剥离**——
        │  剥离会造成数据缺失，损害轨迹完整性。但正则层对图片内容
        │  不可见（一张截图可能泄漏文本层已清洗的一切），因此含图/
        │  含二进制的记录在人工确认门中以专门符号（⚠图）逐条标出，
        │  由用户查看后决定该条数据保留或剔除——图片没有自动防线，
        │  人工确认是它唯一的防线
        ▼
[多层自动脱敏]
  1. Gitleaks 规则集 → API 密钥/令牌/连接串
     （命中即阻断：未逐项处置前禁止导出）
  2. 用户自定义规则（正则或完整字符串）→ 公司名、内网域名等
     （命中即阻断，同上）
  3. 结构感知归一化 → 本地路径、用户名、邮箱、IP
     采用「会话内一致的确定性化名映射」（同一路径/用户名全程映射到
     同一占位符），保留跨轮次连贯性以维持训练价值
     （重点扫描工具调用结果、命令回显、系统提示等高危结构位置）
        ▼
[人工确认门 - 强制]
  分层确认语义：
  - Layer 1/2 命中项：全量枚举，逐项处置（替换/删除/放行），未清零不解锁；
    提供按类型/按规则的一键批量替换——批量替换沿用确定性化名映射
    （同类命中替换为同一占位符），既提效又不破坏数据的训练可用性
  - Layer 3 归一化结果：报告全量统计 + 高危片段抽样人工核对
  - 含图片/二进制记录：⚠ 符号逐条标记，用户查看后决定纳入或剔除
  用户最终签署的是「命中项已全部处置 + 含图记录已逐条确认 + 抽检通过」，
  而非通读全部数据
        ▼
[出口插件 - 用户自选]
  出口① HF 公开数据集通道（v0.1 优先实现）:
      导出统一 schema
      → 本地强制预检：工具内置与 CI 完全相同的扫描（共享三层规则集），
        不通过则拒绝生成 PR —— 因为 PR 一旦创建即公开，且 GitHub 永久
        保留提交历史；CI 扫描的定位是「验证防线」而非「拦截防线」
      → 向社区数据仓库发 GitHub PR
      → CI 自动复扫（应恒为绿）+ 维护者人工 review
      → 合并后批量同步到 HuggingFace 数据集
  出口② API 重放直投（v0.2）:
      环境变量注入 base_url/key → 目标站点用户自配置 + 官方预设
      （DeepSeek 等开源模型厂商）→ 重放脱敏会话 + meta 消息
```

**关键设计决策记录**：
- 社区收集用 **GitHub PR 投稿**而非自建服务器：零基建成本；审查链路全公开可审计（比黑箱服务器检查更取信于社区）；CI 自动扫描即"二次脱敏检查"；PR 是开源社区的母语。注意 PR 通道主要服务开发者贡献者；无 GitHub 账号的非开发者用户当前可走出口②参与，自建服务器收件作为规模化后面向更广人群的 v2 选项。
- 非文本内容（图片/二进制）**标记而非剥离**（推翻评审第一轮的"默认剥离"建议）：剥离保隐私但造成数据缺失，标记 + 逐条人工确认在保住轨迹完整性的同时把图片的防线明确交给人（图片本就没有可行的自动扫描防线）。
- 目标站点（出口②）**用户可自行添加**，官方提供开源模型厂商预设，均使用用户自己的 key。
- 脱敏采取"自动多层清洗 + 导出前强制人工确认"：纯自动必漏、纯人工必死，组合是唯一既可行又可信的方案。
- 化名映射的作用域为**会话内一致、跨会话不一致**（v0.1）：跨会话一致的映射虽能保留多会话项目的连续性，但会使贡献者可被跨会话关联，与重识别风险缓解方向相悖；牺牲连续性换隐私是有意为之的取舍。
- 本地预检与 CI 共用**同一份规则集，且覆盖全部三层**：除 Gitleaks 与自定义规则外，把邮箱/IP 等 Layer 3 类别的正则也纳入共享规则集，使"验证防线"对归一化层同样生效（否则抽检漏掉的邮箱/IP 在下游零校验）。
- **技术栈由 Python 改为 TypeScript/Node**（2026-07-07 复用评估后推翻原决策）：可复用资产全在 TS 生态——omnicross（MIT，npm 已发布）提供 `@omnicross/core` 的 Anthropic↔OpenAI 纯函数转换器（含流式，直接消掉出口②的格式转换层）、`@omnicross/contracts` 的 31 个 provider 预设（即出口②官方预设目录）、cli-launcher 的环境注入配方（模式复用，直连场景自写薄层）。注意 Gitleaks 规则为 Go RE2 正则方言，转译为 JS 正则需加兼容性校验步骤。
- **架构采用 omnicross 已验证形态，分期实施**：core 包（adapter + 脱敏 + 出口插件）+ 本地 daemon + React 审查 UI（daemon 自 serve `/ui`，`npx <tool> ui` 浏览器打开，零 CORS 配置）；Tauri v2 桌面壳推迟到 v0.2+（adopt-or-spawn daemon 模式照搬 omnicross）。GUI 是真需求：人工确认门的命中枚举、批量替换、含图记录逐条预览在终端 UI 下不可用。
- **出口②改为 API 直发而非启动 code CLI**：脱敏会话直接 POST 到目标厂商 `/v1/messages`（或经 converter 转 OpenAI 格式）——CLI 在原方案中只是运载工具，直发链路更短、成本更透明；多轮重放的 token 成本估算仍然必需。
- **elftia 会话解析层以"重新授权或洁净室"方式复用**：elftia（GPLv3）的会话发现/解析代码（`filesystem.ts`、`JsonlParser.ts`、`CliSourceAdapter` 适配器体系，含现成的 Codex/.zst 与 Cursor/SQLite 读取器）耦合极低、质量合适，但 GPLv3 不能直接拷入 MIT 项目；若版权完全属于发起人，抽成独立 MIT 包（elftia 可回头依赖它），否则按其 `JsonlEntry`/`ParsedAgentMessage` 数据模型洁净室重写。

## Open Questions

1. **导出 schema 细节**：发起人已有数据集方案腹稿，需落成文档（字段、粒度、会话/轨迹的切分方式），工具导出格式以此为准。
2. **数据集许可证**：CC-BY-4.0 / ODC-BY / Apache-2.0（数据）待定。
3. **ToS 合规**（⚠️ 必须处置的风险）：会话中包含 Claude 等闭源模型的输出，Anthropic 服务条款禁止用其输出训练竞争模型。公开数据集处于灰色地带，出口②定向投喂则更明确踩线。候选缓解需**按通道分别制定**——注意"剥离模型回复正文"与本项目两大支柱直接冲突：剥离后轨迹不再是轨迹（价值主张受损），且出口②多轮重放必须有 assistant 消息才能重建对话。可行的分通道策略示例：出口①公开数据集剥离或摘要化模型回复正文（接受价值折损换合规），出口②保留完整会话（合规风险由用户知情承担并在工具内明示）。两通道的 ToS 分析结论不同，需在 v0.1 发布前分别决定。
4. **重识别残余风险**：即使脱敏完善，写作风格/话题仍可能重识别用户——是否在贡献流程中加显式知情同意声明？
5. **投喂有效性**（出口②）：厂商数据管线是否会采用重放流量无从验证；meta 消息格式是否需要与厂商沟通约定？
6. HF 组织名 / 数据仓库命名待定。
7. **elftia 模块重新授权确认**：elftia 为 GPLv3，其会话解析层若要以 MIT 复用，需确认相关文件无外部贡献者版权（git blame），否则洁净室重写。

## Success Criteria

- **v0.1 完成的标志**：发起人自己的一批 Claude Code 会话走完「采集→脱敏→人工确认→PR→CI 扫描→合并→同步 HF」全流程，公开数据集 v0.1 上线。
- 工具可被第三方开发者 `npm install` / `npx` 安装并成功导出自己的会话（实现语言选 TypeScript/Node：复用 omnicross MIT 生态与 elftia 会话解析设计，理由见关键设计决策记录；原 Python 决策已推翻）。
- CI 二次扫描在真实 PR 上跑通，且对故意埋入的**金丝雀密钥（无效的假密钥，因测试 PR 本身公开且历史永久）** 100% 拦截。
- 至少 1 位非发起人的社区贡献者成功投稿。

## Next Steps

1. **落地导出 schema**：把发起人腹稿写成 `SCHEMA.md`（对齐出口①与出口②的共用中间格式）。
2. **确认 elftia 相关文件版权归属**（git blame 检查是否有外部贡献者）→ 抽出会话发现/解析层为独立 MIT 包，或按其数据模型（`JsonlEntry`/`ParsedAgentMessage`/`CliSourceAdapter`）洁净室重写。
3. **core 包骨架（TypeScript）+ Claude Code adapter**：读取 `~/.claude/projects/` JSONL，白名单式选择项目/会话（默认只推荐公开 git remote 的项目），标记非文本内容（不剥离），拷贝到隔离工作区。
4. **脱敏层**：集成 Gitleaks 规则集（解析其 TOML 规则、对 JSONL 字段值逐一应用；Go RE2 → JS 正则方言转译需加兼容性校验）+ 用户自定义规则文件（正则/字符串），实现会话内一致的确定性化名映射（路径/用户名/邮箱）。
5. **人工确认门（daemon + React 审查 UI）**：本地 daemon 自 serve `/ui`，Layer 1/2 命中全量枚举、逐项处置、未清零不解锁，支持按类型/规则一键批量替换（沿用确定性化名映射）；Layer 3 统计报告 + 抽检；含图记录 ⚠ 标记逐条预览确认。
6. **社区数据仓库 + CI**：建 GitHub 数据仓库，工具内置与 CI 相同的本地预检（不通过拒绝生成 PR），CI 复扫 + 埋金丝雀密钥验证拦截率，打通 HF 同步脚本。
7. **出口② 直发器（v0.2）**：复用 `@omnicross/core` ApiConverter + `@omnicross/contracts` provider 预设，脱敏会话直接 POST 目标厂商 `/v1/messages`（或转 OpenAI 格式打 `/chat/completions`），不再启动 code CLI；多轮重放 token 成本随轮数近似平方增长——实现前先做一次成本估算。
8. **泄漏应急预案（v0.1 发布前必须就位）**：既然"泄漏一次即摧毁信誉"，必须有 post-publication 处置流程——发现已合并/已同步数据含密钥时：立即从 HF 数据集删除该记录并重新发版 → 数据仓库历史重写（或极端情况下轮换仓库）→ 通知贡献者立刻吊销/轮换泄漏的凭据 → 在仓库公开事故记录。

## The Assignment

**本周做一件事**：用手写脚本（不用等工具成型）把你自己的 1 个真实 Claude Code 会话手工脱敏，按你腹稿中的 schema 整理出来，发给 2-3 位也用 AI 编程的朋友看，问他们两个问题："你愿意用一个工具把你的会话也这样导出吗？"和"看完这份样本，你最担心什么泄漏？"——他们的担忧清单就是脱敏层的第一份需求文档。

## What I noticed about how you think

- 你不是从"我想做个工具"出发，而是从梁文锋的"科技是平权的，而不应该是被垄断的"出发反推出项目——先有价值观再有产品，这种项目往往走得更远。
- 你带着完整的机制腹稿来的："环境变量注入 base_url 与 key，启动 code cli 加载脱敏后的数据，发送一条包含 meta 信息的消息"——这不是模糊的愿望，是已经想清楚传输链路的工程方案。
- 当我推荐 HF 发布时，你没有顺从，而是立刻指出"很多人可能没有 hf 账号，这个要如何处理呢"——用真实用户的摩擦点反驳设计，这是产品直觉。
- 你主动纠正我"目标网站是用户可以自己添加的，我们也会提供一些预设"——在我把你的设计理解窄了的时候，你守住了它的开放性。
