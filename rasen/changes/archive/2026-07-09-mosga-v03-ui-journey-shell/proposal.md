## Why

Slice 1 gave `@mosga/ui` a design system (tokens, primitives, lucide, dark mode) but left the information architecture untouched: the review is still a two-state app (Picker ↔ a 5-tab `ReviewView`) where the user's real journey — pick a session → disposition hits → sign → choose an exit → receipt — is flattened into peer tabs (`blocking`/`nontext`/`l3`/`export`/`submit`) with no visible sense of "which step am I on, and what unlocks the next one." The approved design doc's Approach B (sections **B2 应用外壳** + **B3 四步旅程**) fixes exactly this: an omnicross-style NavRail shell, a persistent 4-step stepper with a live lock badge, a merged disposition workspace, a ceremony-grade signing card, and two equal exit cards. This slice is the structural restructure; it is a prerequisite for slice 3 (the 出口① publish wizard needs the exit-card slot this slice creates).

Behaviour is conserved: disposition, gate, batch, non-text, and 409 semantics do not change. What changes is the container and the flow around them. Unlike slice 1, structural reorganization of the 4 existing test files is expected and allowed — the behavioural *contracts* (gate locked until cleared, disposition calls the daemon, batch delegates, non-text decrements, signing gates the exit) are preserved and re-expressed against the new components, with new tests for the stepper/signing/navigation states.

## What Changes

- **NavRail application shell** (`NavRail` + `AppShell`): left rail (surface-1, w-56) with the "MOSGA" logo + subtitle, two nav items — **贡献** (the review journey) and **设置** (settings) — and a footer showing the daemon address + health. (「历史」is out of scope per the design doc's Later list.)
- **Persistent 4-step stepper + lock badge** (`Stepper`): a always-visible ①选择会话 → ②处置命中 → ③签署确认 → ④选择出口 rail with a right-aligned lock badge in **4 states**: `还差 N 项解锁` (locked) / `已解锁` (cleared, unsigned) / `已签署` (signed) / `已完成` (exit action succeeded). Steps ③ and ④ are gated (③ not enterable until cleared; ④ not until signed).
- **Step ② merged disposition workspace** (`DispositionWorkspace`): the `blocking` / `nontext` / `l3` tabs collapse into a left group nav (密钥命中 N / 自定义规则 N / 图像附件 N / 归一化统计) + a right disposition queue. `FindingsTable` / `NonTextList` / `Layer3View` disposition logic is reused unchanged; batch operations are promoted to queue-top suggestion cards ("aws-access-key × 3 → 一键替换为化名"). 归一化统计 is read-only (stats + spot-check only), produces no disposition, and does not count toward the lock badge — matching the existing gate semantics (blocking + non-text only).
- **Step ③ signing ceremony card** (`SigningCard`): greyed/unenterable until the gate clears; on clear it surfaces a Georgia-titled 「数据捐赠确认」 card with a disposition summary (replace/delete/allow counts, non-text confirm counts, L3 stats + spot-check conclusion), the existing signed-summary checkbox text, and a 「签署并继续」 button. **Signing is client-side state** (as today): a refresh invalidates it (in-memory), and changing any disposition after signing **voids the signature and re-locks step ④** — guarded by a `ConfirmDialog` ("this voids your signature"). The server gate's 409 remains the final backstop.
- **Step ④ dual exit cards** (`ExitCards`): two equal cards. 出口①「公开数据集」is a **readiness-state placeholder** (the 3-step publish wizard is slice 3) — it shows the exit's shape and a disabled/"coming in publish wiring" CTA. 出口②「API 直投」keeps every `SubmitPanel` semantic (target/model/mode, cost estimate, dual acknowledgment, submit) restyled to a card, with the receipt rendered as a **summary card**. A low-key secondary action 「仅导出脱敏文件」preserves the existing export→`ExportPreview` capability. The **receipt view is step ④'s completion state** (stepper all ✓, badge 已完成), not a fifth step.
- **All raw JSON demoted into 「高级」 collapsibles**: `ExportPreview` and the submit receipt no longer show raw JSON as the primary载体 — human-readable summaries are primary; the raw `SanitizedSession` / receipt JSON moves into an expandable Advanced fold.
- **Settings page** (`SettingsPage`): the **three-state theme toggle** (light / dark / system) that replaces slice 1's follow-system-only `theme.ts` bootstrap, plus the daemon address + health and a read-only list of configured provider targets (from the existing `/api/providers`). The data-repository path display and preflight-driven provider-key status are deferred to slice 3 (they need daemon endpoints that slice 3 adds).
- **`dialog` + `confirm-dialog` primitives** ported from omnicross (the slice-1 deferral), adding `@radix-ui/react-dialog`; `dialog` cleaned of the `wallpaper-solid` skin class. `ConfirmDialog` is consumed by the signature-void guard.
- **`useDaemonStatus` hook + `getHealth()` client method**: a small poll of `/api/health` driving the NavRail footer status (additive; no existing behaviour touched).
- **Badge-in-button HTML fix** (slice-1 reviewer Minor): the `Picker` renders a `Badge` (`<div>`) inside a `<button>` (invalid HTML) — switch `Badge` to render as a `<span>` (or accept an `asChild`/`as` element) so it nests validly.

## Capabilities

### New Capabilities

- `ui-journey-shell`: the `@mosga/ui` application shell and journey information architecture — NavRail + settings navigation, the persistent 4-step stepper with its 4-state lock badge, the merged step-② disposition workspace (grouped nav + queue + batch suggestion cards + read-only L3), the step-③ signing ceremony card and its client-side signature lifecycle, the step-④ dual exit cards + secondary export + receipt completion state, the settings page with the three-state theme toggle and daemon status, and the demotion of raw JSON into Advanced folds.

### Modified Capabilities

- `review-ui`: the gate/signing and export-preview requirements are re-presented (gate banner → lock badge + signing ceremony card; raw-JSON export preview → human-readable summary with the JSON demoted to an Advanced fold). The behavioural contracts are unchanged — only the presentation the requirements describe changes.

## Impact

- **Modified package**: `packages/ui/` (`@mosga/ui`) only. No daemon route/config changes (those are slice 3).
- **New files**: `src/components/shell/{NavRail,AppShell,Stepper}.tsx`, `src/components/SettingsPage.tsx`, `src/components/journey/{DispositionWorkspace,SigningCard,ExitCards}.tsx`, `src/components/ui/{dialog,confirm-dialog}.tsx`, `src/lib/useDaemonStatus.ts`, and a small Advanced-fold helper.

- **Tests**: structural reorganization allowed — `smoke`, `FindingsTable`, `GateBanner`→signing/stepper, `ReviewView`→journey; behavioural contracts preserved, new stepper/signing/navigation tests added. Final task = root typecheck + build + vitest green + `rasen validate --strict`.
- **Out of scope** (must not bleed in): any daemon route/config change, the 出口① publish wizard + preflight + data-repo-path display + provider-key status (all slice 3), the 历史 page and i18n framework (Later), bundled fonts / external network assets (Tauri CSP).
