## Problem Statement

当前前端（`packages/ui`，React 18 + Vite + Tailwind，9 组件约 1.1k LOC）是"功能可用的线框"：

- **零设计系统**：Tailwind 配置全默认，无色板、无字体设置（浏览器默认字体）、状态图标全靠 emoji（🔒🔓⚠）。
- **交互 = 表格 + JSON 转储**：导出预览与提交回执直接 `<pre>` 输出原始 JSON（`ExportPreview.tsx:32`、`SubmitPanel.tsx:250`），命中审查是裸 `<table>`——对"贡献者不只是开发者"的目标人群不可用。
- **信息架构掩盖旅程**：审查页 5 个平铺 tab（blocking/nontext/l3/export/submit），但真实用户旅程是线性的（选会话→处置→签署→出口→回执）；"我走到哪一步、还差什么才能解锁"不可见。
- **产品级断点**：出口① 的 HuggingFace PR 发布逻辑在 `packages/publisher` 已完成（`planContribution`/`stageContribution`/`submitContribution`，含强制预检），但**未接入 daemon 路由与 UI**——界面上出口① 止步于"给你一坨 JSON"。

用户原话："当前的前端只能说有个功能，不管是实用性还是美观程度都非常的差，让人没有用下去的欲望。"

## What Makes This Cool

1. **设计系统整体移植而非重画**：omnicross（`E:\AI\ChatAI\Agents\VibeCodingProjects\elftia\elftia\omnicross`，MIT）与 MOSGA 完全同构（Tauri + daemon + React/Tailwind + 本地 `/ui`），其 Claude 风格设计系统——暖象牙 4 级 surface、赤陶橙 accent `hsl(15 56% 52%)`、Georgia display 衬线、13 个 shadcn 式 cva 组件基元、lucide 细线图标、完整 `.dark` 深色模式——可以按文件级移植。MOSGA 本就复用 `@omnicross/core` 生态，两个产品成为"同一家族"。
2. **确认门从横幅升级为仪式**：gate 是本产品的 signature moment（"未清零不解锁"是核心安全语义）。清零后才浮现的衬线字体签署卡，把知情同意从一个 checkbox 变成一次有分量的签署动作——信任感即产品价值。
3. **旅程可见**：常驻步骤条 + 锁徽章实时倒数（"还差 7 项解锁"），处置进度即游戏进度条。
4. **双通道终获对等地位**：出口①/② 并列成两张出口卡——"双通道尊重隐私偏好"是 README 的核心主张，此前 UI 里出口① 根本不存在。

## Constraints

- 个人开源项目，无预算；纯本地渲染（daemon serve `/ui`，无 CORS/CDN 依赖）。
- UI 永远只见脱敏后的 `matchPreview`，重设计不得引入任何展示原始密钥的路径。
- 现有处置逻辑（per-hit/批量 disposition、gate 语义、409 处理）与 4 个测试文件的行为契约保持不变——换容器不换逻辑。
- omnicross 为 MIT，令牌 CSS 与组件基元可直接复制；elftia 主仓库 GPLv3 部分不得引入。
- 出口② 的双重知情确认（ToS + 完整保留）语义已定稿（2026-07-09 决策），只做视觉升级不动语义。

## Premises（已与发起人确认，2026-07-09）

1. 设计系统直接移植 omnicross（令牌 CSS + 组件基元 + lucide），不重新发明。
2. 实用性差的病根是信息架构（5 平铺 tab + JSON 转储），不只是配色；重设计必须让旅程与解锁条件显性化。
3. JSON 转储替换为人类可读摘要卡片；原始 JSON 收进可展开的「高级」折叠。
4. "信任感"是设计第一目标；确认门按仪式设计。
5. 出口① 发布向导遵循 publisher 现有三段式（预检→PR 预览→提交，gh 自动/手动命令兜底），新增 daemon 路由 + 数据仓库路径配置。

## Approaches Considered

### Approach A: 原地换装（minimal viable）
保留两态结构（Picker → 5-tab ReviewView），移植令牌逐个重刷组件，发布做第 6 个 tab。
Effort S-M / Risk Low。**否决理由**：tab 平铺病根未治，"实用性差"的一半抱怨原样保留；发布塞第 6 tab 加剧出口分叉混乱。

### Approach B: 旅程重构（ideal architecture）✅ 已选定
omnicross 式外壳 + 显性四步旅程 + 仪式化签署 + 双出口卡 + HF 发布三步向导。详见下节。
Effort L / Risk Med。

### Approach C: 契约签署隐喻（creative）
会话全文档视图 + 命中即内联批注 + 文末签名区。隐喻与知情同意语义完美同构，但需新写会话正文渲染器（当前 UI 不渲染会话全文），长会话性能风险，工期 ≈ A 的 4 倍。**否决理由**：v0.x 性价比不足；其仪式感精华（签署卡）已被 B 的第③步以 ~10% 成本吸收。

## Recommended Approach

**Approach B：旅程重构**。线框图：[`frontend-ui-redesign-wireframe.html`](frontend-ui-redesign-wireframe.html)（3 屏）。

### B1. 设计系统层（新增 `packages/ui` 令牌基建）
- 移植 omnicross `packages/ui/src/index.css` 的 `:root`/`.dark` 语义令牌（4 级 surface、text-strong/muted/subtle、accent `15 56% 52%`、success/warning/danger、radius）与 `tailwind.config` 的 colors/fontFamily/fontSize/borderRadius 扩展。
- 移植组件基元（按需，预计 8 个起步）：button（7 变体 cva）、badge、dialog、confirm-dialog、input、select、switch、tooltip；依赖新增 `class-variance-authority`、`lucide-react`、`clsx`/`tailwind-merge`（cn util）。
- emoji 状态图标全部替换为 lucide（Lock/Unlock/AlertTriangle/CheckCircle 等，strokeWidth 1.5）。
- 字体：Inter（system-ui 栈兜底）正文 + Georgia display 标题 + 等宽栈（预览/命中片段）。深色模式随令牌免费获得（`.dark` class 切换）；切换机制默认跟随系统，设置页是否加三态开关见 Open Questions 1——B2 设置页的「深浅模式」项以该问题的结论为准。

### B2. 应用外壳
- 左侧 NavRail（w-56，surface-1）：Logo「MOSGA」+ 副标语；导航项（v0.x 仅两项）：**贡献**（主流程）/ **设置**（数据仓库路径只读展示 + 重启指引、provider key 配置状态只读展示、深浅模式）；页脚 daemon 地址 + 健康状态。（「历史」页裁出本次范围，见 Later 清单。）
- 内容区顶部**常驻步骤条**：①选择会话 → ②处置命中 → ③签署确认 → ④选择出口；右端**锁徽章**实时显示 `还差 N 项解锁` / `已解锁` / `已签署` / `已完成`（数据源：`还差 N`/`已解锁` 来自现有 `/gate` 端点；`已签署` 来自客户端签署态——签署在 v0.x 是客户端状态，见 B3 ③；`已完成` 为第④步出口动作成功后的完成态）。

### B3. 四步旅程（重组现有组件，处置逻辑不动）
- **① 选择会话**：Picker 升级为卡片列表（项目卡带 recommended 徽章语义不变；"显示全部项目"默认关闭的隐私默认不变）；扫描中 loading 态、空项目/空会话空态设计。
- **② 处置命中**：blocking/nontext/L3 三 tab 合并为「左侧分组导航（密钥命中 N / 自定义规则 N / 图像附件 N / 归一化统计）+ 右侧处置队列」工作区。FindingsTable/NonTextList/Layer3View 的处置逻辑与批量操作原样保留，只换容器与视觉；批量操作提升为队列顶部的建议卡（"aws-access-key × 3 → 一键替换为化名"）。每次处置，锁徽章倒数。「归一化统计」分组为**只读**（统计报告 + 抽检），不产生处置项、不参与锁徽章倒数（与现有 gate 语义一致：只数 blocking + non-text）。
- **③ 签署确认**：命中清零前该步灰显不可入；清零后浮现**签署卡**（Georgia 衬线标题「数据捐赠确认」+ 处置统计摘要：替换/删除/放行计数、非文本确认计数、L3 统计与抽检结论 + 现有 signed-summary checkbox 文案 + 「签署并继续」主按钮）。签署即解锁第④步。**签署生命周期**（v0.x）：签署状态保持客户端态（与现 `GateBanner` 一致，刷新即失效需重签）；签署后若返回②修改任何处置，签署立即作废、④ 重新锁定（服务端 gate 的 409 仍是最终防线，UI 状态只是引导层）。
- **④ 选择出口**：两张对等出口卡 + 一个次要动作。出口①「公开数据集」→ 3 步子向导：**预检**（跑 `planContribution`；同步扫描需 pending 态 + 超时提示；`PublishRefusedError` 时按**规则粒度**展示阻断原因——预检命中是对导出字节的复扫，无法逐项映射回②的处置队列，故显示"规则 X × N 处阻断"并提供「回到② 查看该规则分组」的跳转）→ **预览**（PR 标题/正文、文件清单、分支名；markdown 渲染方案见 Open Questions）→ **提交**（无论 gh 与否先 `stageContribution` 落盘；`ghAvailable` 且已登录时一键 `submitContribution`，否则展示已落盘文件位置 + 手动命令序列（注意 `plan.commands` 末条是 `gh pr create`，无 gh 用户的真正兜底是 `git push` + 浏览器打开 compare URL 开 PR，向导需生成该 URL）+ 逐条复制按钮）。出口②：现 SubmitPanel 全部语义保留（成本估算、双 acknowledgment），回执改摘要卡片。次要动作「仅导出脱敏文件」：保留现有 export 能力（下载/复制 stamped JSON），以低调链接形式存在，点开即现 ExportPreview 的摘要卡 + 「高级」折叠。**回执视图**是第④步的完成态（步骤条全 ✓ + 徽章「已完成」，不是第五步）：出口徽章 + 关键字段摘要 + 「高级：原始 JSON」折叠。

### B4. daemon 新增（出口① 接线）
- 路由（gate 未解锁一律 409，镜像现有 `/export`、`/submit` 的处理方式）：
  - `POST /api/reviews/:id/publish/plan` — 导出 + 强制预检 + 计划（纯内存，不落盘）。返回 ContributionPlan 的 **UI 安全子集，字段显式枚举**：`branch`、`targetBranch`、`recordPath`、`provenancePath`、`prTitle`、`prBody`、`commitMessage`、`recordCount`、`ghAvailable`、`stagedFiles`、`commands`、`provenance`（版本戳）、`engine`，外加 daemon 派生的 `compareUrl`（在 clone 内 `git remote get-url origin` 推导出 GitHub compare URL，供 gh-free 兜底路径用——`ContributionPlan` 本身不含远端 URL）；**排除** `record` 的序列化字节（体积大且预览无需，摘要以字节数 + 内容哈希代替）。
  - `POST /api/reviews/:id/publish/stage` — 将计划落盘到数据仓库 clone（写记录 + provenance sidecar + PR body 文件、建分支）。gh 缺席用户走到这一步为止，UI 展示落盘位置 + 手动命令 + compare URL。
  - `POST /api/reviews/:id/publish/submit` — stage（若未 stage）+ `submitContribution`。
  - **错误分类须与现有 `/submit` 同标准**（参照 ConsentError/NotStamped/KeyNotConfigured 的建模）：`precheck_refused`（含规则聚合明细）、`repo_dirty`（工作区不干净）、`branch_exists`（分支名确定性派生，重试必撞——响应中带既有分支名与处置建议）、`gh_unauthenticated`（gh 在但未登录）、`push_rejected`。
  - **执行模型**：git/gh 子进程必须异步执行（不得阻塞 daemon 事件循环）。注意 publisher 现有 `CommandRunner` 接口是同步的（`runner.ts`），满足此要求需要**给 `packages/publisher` 加异步 CommandRunner 变体或把整个 stage/submit 调用卸载到 worker 线程**——这是切片 3 的一项显式任务，是"换容器不换逻辑"约束的一个例外（接口加宽，行为不变，同步接口保留供 CLI/测试）。同一时刻只允许一个 in-flight publish（简单互斥锁），UI 显示进行中状态。
  - **stage 状态模型**：daemon 在内存中按 review 记录 publish 状态（`staged` 标志 + 分支名）；`submit` 的"若未 stage 则先 stage"以该内存标志为准。首次 stage 即遇 `branch_exists`（无内存标志）判定为上一次尝试的残留分支，错误响应附带"删除该分支重试或手动继续"的指引，不做自动清理。
- 配置：daemon options 新增 `dataRepoPath`，**严格沿用 `providerKeyConfigPath` 的信任模型——启动时服务端配置，永不经 HTTP 写入**。设置页对它是**只读展示**（当前值/未配置 + "以 `--data-repo <path>` 重启 daemon"的指引文案），不提供编辑框。
- `GET /api/publish/preflight`：返回 `{ dataRepoConfigured, gitAvailable, ghAvailable, ghAuthenticated, repoClean }`（gh 登录态用 `gh auth status` 探测），驱动出口① 卡片四种状态：就绪 / 需配置（只读指引）/ gh 未登录 / 缺依赖。

## Open Questions

3. 出口① 预览中 PR body 的 markdown 渲染：引入轻量渲染器（如 marked）还是先用带样式的 `<pre>`？（安全上 PR body 由本地生成，风险低；建议切片 3 先 `<pre>`，渲染器视观感决定。）
4. 回执持久化（为将来的历史页铺路）：daemon 加 `~/.mosga/receipts.jsonl` 轻量落盘，还是保持内存态？不阻塞本次三个切片，但切片 3 的回执视图若顺手落盘成本很低。

## Success Criteria

- 走完「选会话→处置→签署→出口②直投（或出口① dry-run 到 PR 预览）」全程，任一时刻能在 3 秒内答出"我在哪一步、还差什么"（步骤条 + 锁徽章）。
- 界面上不再存在任何裸 `<pre>` JSON 作为主要信息载体（全部降级为「高级」折叠）。
- 出口① 在 UI 内可完成：预检 → PR 预览 → gh 一键提交（或手动命令）；`PublishRefusedError` 有可读的阻断提示。
- 视觉与 omnicross 同源可辨（同令牌、同组件基元、同图标语言），深浅两模式均可用。
- 现有 4 个测试文件行为契约不回退；处置/gate/409 语义零变化。
